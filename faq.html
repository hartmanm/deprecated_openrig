Frequently Asked Questions



For reference: PL,CC,MC,FS,EH

PL is Power Limit (in watts)
CC is Core Overclock (offset from stock clock, use 0 for stock clock)
MC is Memory Overclock (offset from stock clock, use 0 for stock clock)
FS is Fan Speed (in percent, use 49 or lower for automatic/dynamic speed)
EH is Expected Hashrate (this is the hashrate you expect from the rig)

Ensure you set EH low until you are sure what to expect, otherwise you will send your rig into a boot loop.

Q: Is it possible to see the console screen?

A: By clicking the console button you will display the rig Public IP, Private IP, MAC address and the last 10 lines of the console received from the rig. Click it again to close the display.

The watchdog will send hashrate and console updates every 60 seconds.

If a rig fails to update its hashrate multiple times consecutively it will be added to the drop list and a large red button will appear below the LOG. This button will have the rig number and the time the drop was detected. It can be dismissed by clicking.
When a rig is added to the droplist, it is annotated in the LOG in addition to creating a red button.

Q: When I update settings do I have to reboot each rig to enable the changes or is it automatic?

A: Every rig checks for configuration updates every 60 seconds. If there is any modification it will implement it automatically. There is no need to manually reboot. If you only want to restart the mining client and watchdog click the reclient button. It will stop the mining process and watchdog and reinitialize them.

Q: What does the orange delete button do?

A: delete when clicked allows for the deletion of the last created group or rig. When clicking it you should see a red delete button appear. Because the intent is for the interface to work for smartphones and tablets in addition to PCs; the additional confirmation guards against most accidental deletions. As groups and rigs can easily be repurposed the only reason to delete (a group or rig) should be when it is no longer needed.

Q: For Ethash and x16r how can I enter the worker name?

A: There is no way to modify workernames using COIN selections, they are hardcoded to the rig number. If you set the COIN to CLIENT you can use any arguments you wish.

Q: Can I just add something like this? stratum+tcp://rvn.suprnova.cc:6667 -u user.miner22 -p x

A: Yes, you can use a normal execution line with the CLIENT_ARGS field. Set COIN to CLIENT, choose a mining client from the CLIENT dropdown, enter your client arguments in the CLIENT_ARGS field, and finally set your OC with the CLIENT_OC field.

Q: Is the H/s the current hash or average?

A: It is the last response from the Rig. The Rigs do not maintain a connection to the back-end, instead, they interact with an API by sending and receiving messages. When using email updates or telegram; all changes will trigger an email or telegram update.

Q: I assume if I see reload 1 then 2 then 3 something is wrong and it is automatically resetting.

A: Again check the console and emails for more information. The watchdog will try stopping and restarting the mining process until it exceeds 3 attempted restarts within a threshold; then it will reboot. If there is a configuration problem that is not corrected this will occur in a loop.

Q: How do you check the IP of each rig? Is there an option? My hosting provider will start putting all of the rigs on openrig, but they won't check the ip of each when they convert.

A: By clicking the console button you will display the rig Public IP, Private IP, and MAC address.
In addition, the IP (full IP) and Mac (last 3 characters) are sent with each email update or telegram, in addition to GPU_COUNT, RIG#, VERSION# and the RIG LOG.
RIG LOG is stored in memory on the rig (is cleared on reboot) and will append new events such as:
new settings detected and implemented
low utilization triggering a restart of the mining client
low hashrate triggering a restart of the client
rebooting due to successive low utilization or hashrate
Startup, update, or initial reboot (_init)
All RIG LOG entries have a full date and time stamp.
